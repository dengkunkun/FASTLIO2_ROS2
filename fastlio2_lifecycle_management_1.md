## 问题 1：Nav2 VoxelLayer + 静态点云 → 合成实时点云地图？

**答案：不能。**

```
Nav2 VoxelLayer 的本质：
  ┌─────────────────────────────┐
  │   2.5D 栅格（不是3D点云）    │
  │   每个 (x,y) cell 存的是    │
  │   "占据/空闲" 的列状体素    │
  │   没有精确的3D几何信息       │
  │   只用于导航避障             │
  └─────────────────────────────┘

你需要的：
  ┌─────────────────────────────┐
  │   精确3D点云                 │
  │   每个点有精确的 (x,y,z)    │
  │   用于相机反投影计算着火位置  │
  └─────────────────────────────┘
```

VoxelLayer 是 costmap 工具，分辨率粗（通常 5-10cm 体素），没有点级精度，无法用于相机-点云联合定位计算。

### 关键：自定义"点云更新节点"的实现思路

这是整个方案中唯一需要新写的核心模块，但工作量不大：

```cpp name=map_updater_node.cpp
// 伪代码：轻量级点云增量更新节点
class MapUpdaterNode : public rclcpp::Node
{
    // 加载初始地图
    pcl::PointCloud<pcl::PointXYZI>::Ptr global_map_;
    KD_TREE<PointType> ikdtree_;  // 复用 FASTLIO2 的 ikd-tree

    void init()
    {
        // 启动时加载 PCD
        pcl::io::loadPCDFile("global_map.pcd", *global_map_);
        ikdtree_.Build(global_map_->points);
    }

    // 每帧回调：当前 scan + 当前位姿 → 增量更新地图
    void scanCallback(const PointCloud2::SharedPtr scan_msg)
    {
        // 1. 将 scan 转换到 map 坐标系（用定位结果的 TF）
        auto world_scan = transformToMap(scan_msg);

        // 2. 新增点：scan 中的点，在 ikdtree 中无近邻 → 新增
        for (auto& pt : world_scan->points)
        {
            PointVec nearest;
            ikdtree_.Nearest_Search(pt, 1, nearest, distances);
            if (distances[0] > 0.15)  // 15cm 内无已有点
            {
                ikdtree_.Add_Points({pt});  // 增量添加
            }
        }

        // 3. 可选：删除点（射线追踪清除）
        //    对每条射线，如果穿过了已有点但没有遇到障碍
        //    → 说明该区域已变空 → 删除这些点
        //    （这是最复杂的部分，可简化或不做）

        // 4. 定期发布更新后的地图
        publishUpdatedMap();

        // 5. 定期保存到磁盘（增量 PCD）
        if (shouldSave())
            pcl::io::savePCDFileBinary("global_map_updated.pcd", ...);
    }
};
```

### 为什么这个方案比改造 RTAB-Map 或 FASTLIO2 更好

| 维度 | 改造 RTAB-Map | 改造 FASTLIO2 | 混合方案 |
|------|-------------|-------------|---------|
| 开发工作量 | 小（调参为主） | 极大（补全缺失功能） | 中等（写1个新节点） |
| 内存占用 | 高（SQLite+图优化+OctoMap） | 低 | 低（ikd-tree ~500MB） |
| 磁盘占用 | 高（DB 持续增长） | 低（PCD 文件） | 低（PCD 文件） |
| 建图精度 | 中等（ICP 无 IMU 紧耦合） | 高（IESKF） | 高 |
| 定位稳定性 | 好 | 差（你已验证） | 好（FAST-LIO odom + 周期 ICP 校正） |
| 地图更新 | 慢+有 bug | 无 | 可控 |
| 维护复杂度 | 高（RTAB-Map 参数地狱） | 中 | 低 |
| 长时间运行 | OOM 风险 | 你说会报错 | 稳定（各模块独立） |

## 关于"倾斜安装 LiDAR/IMU 导致 FASTLIO2 长时间报错"

这个问题大概率是外参标定问题，不是 FASTLIO2 的 bug：

```yaml name=fastlio2_config.yaml
# FASTLIO2 的 LiDAR-IMU 外参配置
# 如果 LiDAR 倾斜安装，必须准确设置这些参数

# LiDAR 到 IMU 的旋转矩阵（关键！）
# 如果 LiDAR 向下倾斜 30 度绕 Y 轴：
extrinsic_R:
  - 0.866, 0, 0.5
  - 0, 1, 0
  - -0.5, 0, 0.866

extrinsic_T:
  - 0.0, 0.0, 0.0    # LiDAR 到 IMU 的平移
```

长时间运行报错通常是因为：
1. 外参不准 → IMU 预积分与 LiDAR 匹配矛盾 → 状态发散
2. 重力方向错误 → 倾斜安装时 IMU 初始化阶段估计的重力方向有偏差

## 关于火灾检测的 2D→3D 反投影

这部分与 SLAM 框架无关，核心是：

```
已知：
  - 相机内参 K (3×3)
  - 相机外参 T_map_camera (4×4)，来自 TF 树
  - 火焰像素坐标 (u, v)
  - 全局点云地图 M

计算：
  1. 像素坐标 → 射线方向: ray = K^(-1) × [u, v, 1]^T
  2. 射线转到 map 坐标系: ray_map = R_camera × ray
  3. 射线起点: origin = t_camera (相机在 map 中的位置)
  4. 射线与点云求交: 沿 ray_map 方向找最近的点云交点
  5. 交点就是 3D 着火位置
```

这个计算 **只需要一个静态（或准静态）点云地图 + 相机的实时位姿**，不需要 10Hz 的地图更新。
